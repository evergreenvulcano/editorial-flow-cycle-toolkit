# Contributing

Thank you for your interest in contributing to the Editorial Flow Cycle Toolkit.

This is a documentation-first, v0 project. Contributions are welcome within the scope defined below.

## Scope of v0

V0 is a repo of documents, templates, schemas and prompts. It does not contain application code, a database or a publishing pipeline.

Contributions should stay within this scope. Do not add:

- application code unless explicitly requested
- database schemas or migration files
- publishing automation scripts
- dashboard or analytics tooling
- backlog or kanban machinery
- dependencies or package managers (unless a schema validation script is explicitly approved)

## What contributions are welcome

- Corrections to documentation (clarity, accuracy, completeness)
- Improvements to templates (better structure, clearer prompts, missing fields)
- Additions to examples (new neutral examples; examples must use neutral fictional material — no private names, internal terminology or unpublished source references)
- Improvements to schemas (correctness, completeness, alignment with documentation)
- Improvements to prompts (better instructions, clearer constraints, fewer loopholes)
- New prompts for operations described in the cycle model but not yet covered

## How to contribute

1. Open an issue describing the change you want to make and why.
2. Wait for acknowledgment before submitting a pull request.
3. Keep changes small and focused. One change per pull request.
4. Use the existing file naming conventions.
5. Write in plain Markdown. Match the tone of existing documents: precise, collaborative, implementation-ready.

## Language guidelines

- Use "editorial flow cycle" rather than "pipeline" in user-facing content.
- Use "gated execution" when explaining that the cycle can run.
- Use "cycle report" as the default output term.
- Use "reinjection note" for reusable learning.
- Use "public-bound candidate" only when describing a candidate for public-facing preparation, not publication.
- Use "repo/public surface proposal" for possible repo representation.

## Examples must use neutral material

All examples must use neutral, fictional material. Acceptable descriptions include:

- "draft essay"
- "research note"
- "landing page candidate"
- "design memo"
- "strategic brief"

Do not use private names, invented lore, internal project names or references to real unpublished work.

## Anti-scope

Do not add to the repo:

- private project names or references
- local file paths from any specific system
- internal terminology not defined in the repo itself
- claims about a specific organization
- automatic publishing scripts
- growth, content or SEO framing

## Code of conduct

Be direct and constructive. This project values precision over politeness, but also values clarity over cleverness.

Disagreements about scope should be resolved by referring to the principles in [`docs/concept.md`](docs/concept.md).
